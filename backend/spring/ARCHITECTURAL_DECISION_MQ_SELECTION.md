# Architecture Decision Record (ADR): 비동기 메시징 솔루션 비교 및 NATS 최종 선정

본 문서는 **지식 검수 및 요약 시스템(ai-pilot)**의 분산 배포 환경(k3s)에서 Spring Boot(Orchestrator)와 Go(k3s 컨트롤러 및 비동기 워커) 간의 이벤트 기반 워크플로우를 설계하기 위해, 주요 메시징 솔루션들을 비교 분석하고 **NATS**를 최종 백본으로 선정한 기술적 배경을 정리한 아키텍처 의사결정 문서입니다.

---

## 1. 아키텍처 컨텍스트 (Context)
* **다단계 상태 머신**: 본 시스템의 지식 가공 파이프라인은 여러 단계(`DRAFT` ➔ `VERIFYING` ➔ `FORMATTING` ➔ `REVIEW_READY` ➔ `PUBLISHING` ➔ `PUBLISHED`)에 걸쳐 점진적으로 상태가 전이되는 구조를 가짐.
* **이종 언어 협업 (Spring & Go)**: 전체 조율(Orchestration)은 Spring Boot가 담당하고, 쿠버네티스(k3s) 리소스 제어 및 실질적인 가벼운 연산 API는 Go(Golang) 워커가 처리하는 마이크로서비스 지향 설계 예정.
* **k3s(경량 쿠버네티스) 타겟 배포**: 저사양 가상 머신(VM) 및 소형 노드로 구성된 k3s 환경에 배포되어야 하므로 인프라 리소스 소모를 극도로 최소화해야 함.
* **질문 제기**: 단순한 일감 밀어주기식 **작업 큐(Job Queue)**로 갈 것인가, 아니면 전체 시스템의 상태 흐름을 지배하는 **이벤트 기반 워크플로우(Event-Driven Workflow)**로 갈 것인가?

---

## 2. 메시징 솔루션 후보군 (Candidates)
1. **Redis (Streams / List)**: 메모리 데이터 스토어 기반의 경량 대기열
2. **RabbitMQ**: AMQP 표준 기반의 전통적 메시지 브로커
3. **NATS (with JetStream)**: Go 기반의 클라우드 네이티브 초경량 이벤트 스트림
4. **Apache Kafka**: 대규모 실시간 분산 이벤트 스트리밍 플랫폼

---

## 3. 메시징 솔루션 비교 분석표 (Comparison Matrix)

| 비교 항목 | Redis (Streams) | RabbitMQ | NATS (JetStream) | Apache Kafka |
| :--- | :--- | :--- | :--- | :--- |
| **패러다임 분류** | 다목적 스토리지 + 작업 큐 | 전통적 작업 브로커 | 이벤트 기반 워크플로우 백본 | 초대용량 로그 스트리밍 |
| **k3s 인프라 무게** | **매우 가벼움** (~30MB) | 보통 (~150MB) | **극도로 가벼움** (~15MB) | 극도로 무거움 (수 GB 필요) |
| **Go 생태계 친화도** | 좋음 (go-redis) | 보통 (streadway/amqp) | **최상 (Native Go 개발)** | 보통 (confluent-kafka) |
| **지연 시간 (Latency)** | 낮음 | 보통 | **극도로 낮음 (Low-latency)** | 보통 ~ 낮음 |
| **실패 재처리 / DLQ** | 커스텀 구현 필요 | **매우 쉬움 (기본 제공)** | 쉬움 (JetStream 내장) | 복급 제어 직접 설계 필요 |
| **주요 용도** | MVP 빌드, 데이터 캐싱 겸용 | 엔터프라이즈 업무용 분배 | 클라우드 네이티브 이벤트 전송 | 대규모 로그/빅데이터 수집 |

## 4. 후보군별 상세 평가

### 🔴 Redis
* **평가**: 스토리지와 대기열의 특성을 동시에 가져갈 수 있어, 데이터 캐싱이나 임시 상태 보관이 병행되어야 하는 단일 API 서버 구조에 최고의 MVP 솔루션입니다.
* **한계**: 서비스 간의 느슨한 결합(Loosely Coupling)을 지원하기 위한 브로드캐스팅이나 복잡한 라우팅에 제약이 있으며, 다단계 파이프라인에서 이벤트 기반 전이를 구현하기에 구조가 무거워질 수 있습니다.

### 🔵 RabbitMQ
* **평가**: AMQP 표준 프로토콜을 사용하고 복잡한 라우팅 규칙(Exchange 유형)과 DLQ(Dead Letter Queue)를 기본 제공하여 금융, 기업 등 전통적 분산 비동기 처리에서 강력합니다.
* **한계**: Erlang 가상 머신 기반으로 동작하여 NATS나 Redis에 비해 기동 리소스를 더 차지하며, k3s 소형 인프라에 가볍게 Pod로 얹기에는 오버헤드가 있습니다.

### 🟢 NATS (최종 선정 🏆)
* **평가**: Go 언어로 개발된 클라우드 네이티브 메시징 툴로, 컨테이너 크기 10MB 미만, 기동 메모리 15MB 수준이라는 **혁신적인 가벼움**을 자랑합니다.
* **장점**: Go 언어의 고루틴 및 채널과 완벽히 유기적으로 결합되며, 이벤트 스트림 유실 방지 기능(`JetStream`)과 양방향 통신(`Request-Response`)을 동시에 지원하여 다단계 상태 전이 워크플로우에 최적화되어 있습니다.

### ⚫ Apache Kafka
* **평가**: 분산 환경에서의 완벽한 순서 보장과 디스크 스토리지 기반의 대규모 로그 보존이 강점인 거인입니다.
* **한계**: k3s와 같은 엣지/경량 쿠버네티스 노드에서는 기동 자체만으로 리소스를 전멸시킬 수 있어 **배포 부적합** 판정을 내렸습니다.

## 5. NATS 최종 선택 사유 (Architectural Decision)

### 1) "Job Queue" ➔ "Event-driven Workflow"로의 패러다임 전환
단순히 "Spring이 던진 일감을 Go 워커가 처리하고 끝내는 구조(Job Queue)"라면 Redis나 데이터베이스 폴링으로도 충분합니다. 
하지만 본 시스템은 지식 상태가 단계적으로 전이되는 **마이크로서비스 이벤트 분산 환경**을 목표로 합니다. NATS는 각 마이크로서비스(Spring 및 Go)가 서로의 네트워크 정보를 몰라도 `knowledge.draft.created`, `knowledge.verified`와 같은 이벤트 주제를 주고받으며 리액티브하게 연동되는 백본 역할을 완전히 책임집니다.

### 2) Go & k3s 생태계 최적화 (Cloud-Native Synergy)
* **Go 친화도**: Go로 작성할 k3s 오퍼레이터(Operator) 및 Go API 서버가 NATS Go SDK를 사용하여 리소스를 매우 정교하고 안전하게 활용할 수 있습니다.
* **k3s 초경량 Pod 리소스**: 메모리가 한정된 k3s 환경에서 Pod 하나당 15MB 내외의 메모리만 점유하므로 노드 스케일 오버헤드를 극도로 아낄 수 있습니다.
* **RPC 패턴 무상 지원**: 별도의 HTTP 클라이언트나 gRPC 설정 없이, NATS의 `Request-Response` 메시징 패턴을 활용해 비동기 제어 명령과 결과를 하나의 이벤트 채널 안에서 양방향 처리 가능합니다.

---

## 6. 구현 로드맵 (Roadmap)
1. **1단계 (현재 완료)**: Spring Boot 백엔드 애플리케이션 내부에서 세마포어(apiThrottle)와 지수 백오프+Jitter가 포함된 리트라이 처리 코어 구조 안정화.
2. **2단계 (Go API 도입 시점)**: k3s 클러스터 내에 NATS를 띄우고, Spring Boot(Publisher)와 Go Worker(Subscriber) 간에 상태 변화 이벤트를 통해 비동기 가공을 격리 처리하는 이벤트 기반 워크플로우 통합.
